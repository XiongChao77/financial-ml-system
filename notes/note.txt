# Market Understanding

Market states can be modeled in two ways:

1. First distinguish whether to trade or not (`trade vs non-trade`), then decide the direction (`long vs short`).
2. Directly focus on direction and split samples into `long vs other` and `short vs other`.
   An additional filter can be added outside the model to determine `trade vs non-trade`.

From subjective experience, the second approach better matches the actual market structure, because upward and downward moves are often driven by different mechanisms, while “other” usually represents no signal or noise rather than an independent modelable category.

***********************Pitfalls Encountered*************************

Indicator calculation:

`FeatureCandle` / `FeatureKdj` / `FeatureCFM` / `MFI` may become `0` or `NaN` during extreme market conditions and need to be converted into meaningful values.

Normalization: normalize a group of features based on the same base in order to preserve relative characteristics. However, previously both `mu_base` and `sigma_base` used values from the base. `mu` must use the current feature’s mean; otherwise, the mean will not be zero. Using `sigma_base` also loses part of the statistical distribution characteristics.

For absolute-value indicators such as price and volume, using Z-score normalization will produce negative values, which is clearly not expected.

Model prediction signals cannot simply overwrite previous signals. Avoiding repeated predictions is actually the simplest and most effective approach.

The stricter the samples used during training, the better the recall and precision tend to be.

A data alignment issue occurred: the backtest predictions and label indices were offset, making several days of test results invalid.

The input to the model must be relative information, and label filtering can only use relative information as well. Absolute-value information such as `min_threshold` severely affects learning results, so the threshold needs to be normalized by volatility.

Because the optimization objective during training is not directly profit, fluctuations can be large. How to reduce this gap is an important problem.

After linking the loss function with returns/probabilities, the reference value of the F1 metric drops significantly, and Loss must become the primary metric. Training should use loss, but the final selection of specific indicators should use evaluation methods closer to the target rather than simply choosing the best loss. Loss is only used to improve the training method.

Volatility was used incorrectly. When labeling changes that exceed a multiple of the volatility benchmark as positive/negative, if the volatility calculation window is too short, it will reduce market prediction quality.

Volatility solves three problems:

- Adaptation across different assets
- Long-term regime adaptation
- Short-term state adaptation

If the system falls into hyperparameter search, it cannot continue evolving. Most parameters must be fixed, and model capability should be treated as the foundation.

After correlation analysis and bucket analysis, metrics such as F1, accuracy, and recall are negatively correlated with returns! `precision_weighted` is positively correlated. A new evaluation method needs to be found.

A very serious issue: the model’s F1 metric is negatively correlated with strategy results, making optimization very difficult. How to design the loss function is a major problem.

## 📊 Basic Price and Time Information

`open_time`: The start time of the candlestick/K-line, in millisecond timestamp format.

`open`: The opening price of the period.

`high`: The highest price of the period.

`low`: The lowest price of the period.

`close`: The closing price of the period.

`close_time`: The end time of the candlestick/K-line, in millisecond timestamp format.

## 📈 Volume-Related Fields

`volume`: The trading volume during the period, measured in the base asset of the trading pair, such as BTC in BTCUSDT.

`quote_asset_volume`: The traded amount during the period, measured in the quote asset of the trading pair, such as USDT in BTCUSDT.

Formula relationship:

```text
quote_asset_volume ≈ volume × average traded price
```

## 🔄 Trade Statistics

`number_of_trades`: The number of trades during the period, meaning successfully matched orders, without distinguishing buy or sell direction.

`taker_buy_base_volume`: The base-asset volume actively bought by takers.

Example: In BTCUSDT, if taker buy orders are filled for 5 BTC, it is counted here.

`taker_buy_quote_volume`: The quote-asset amount corresponding to taker active buying.

Example: If a taker buys 5 BTC and spends 100,000 USDT, this value is 100,000.

| Metric | Meaning | Explanation |
| --- | --- | --- |
| **precision** | Among samples predicted as this class, the proportion that truly belongs to this class | `precision = TP / (TP + FP)`<br>→ How “accurate” the predictions are |
| **recall** | Among the true samples of this class, the proportion correctly predicted by the model | `recall = TP / (TP + FN)`<br>→ How “complete” the predictions are |
| **f1-score** | Harmonic mean of precision and recall | $F1 = \frac{2\cdot P \cdot R}{P+R}$<br>→ Balances “accuracy” and “coverage” |
| **support** | Number of samples of this class in the test set | Sample base, useful for understanding class size |

| Name | Meaning | Explanation |
| --- | --- | --- |
| **accuracy** | Proportion of correctly predicted samples among all samples | $\frac{\text{All correct predictions}}{\text{Total samples}}$ |
| **macro avg** | Calculate precision/recall/F1 for each class first, then take an **equal-weighted average** across classes | Ignores class-size differences; useful for checking fairness toward small classes |
| **weighted avg** | Calculate precision/recall/F1 for each class first, then take a weighted average based on support, i.e. sample count | Classes with more samples have greater influence; closer to overall performance |

*******************************Data Mining*********************************************

Factor correlation analysis:

Use normalized data for correlation analysis. A large amount of classic literature and modern deep-learning applications in finance have directly or indirectly confirmed that normalization, differencing, or log-return transformation is a necessary prerequisite for effectively measuring factor predictive power.

Granger & Newbold (1974), “Spurious regressions in econometrics”: this Nobel-level paper laid the foundation. It proved that if non-stationary time series, such as raw prices, are used directly for regression or correlation analysis, even two completely unrelated variables may show statistically significant t-statistics. Conclusion: the data must be made stationary through differencing or normalization before the significance of statistics is credible.

Marcos López de Prado (2018), “Advances in Financial Machine Learning”: this is a bible of modern quantitative ML. López de Prado dedicates a chapter to “Fractional Differentiation.” He clearly points out that standard machine-learning models cannot handle non-stationary data, such as raw prices. The data must be transformed into stationary series, similar to normalization or returns, in order to preserve predictive power, i.e. memory. He emphasizes that if input features are not properly stationarized, correlation analysis is meaningless.

*******************************Strategy Validity Verification*********************************************

## 1. Walk-Forward Analysis — Most Recommended

Since the data cannot be split into two large blocks, split it into many small blocks instead to simulate the real passage of time. This is the most standard approach in quantitative trading.

Method:

Set a training window, such as 6 months, and a test window, such as 1 month.

Step 1: Use data from months 1–6 to optimize `thresh`, obtain the best parameter, such as 0.5, and then test it in month 7.

Step 2: Roll forward! Use data from months 2–7 to re-optimize `thresh`, which may become 0.55 this time, and then test it in month 8.

Step 3: Continue rolling forward...

Advantages:

Dynamic adaptation: your parameters change with the market, which matches live-trading logic, because strategies are adjusted periodically.

High data utilization: all data is used both as training data and test data at different points in time.

Out-of-sample validation: you always use “past” data to derive parameters and then run them on the “future” month, without looking ahead.

## 2. Cross-Timeframe Validation

Apply the filtered strategy to different timeframes and check whether it remains effective.

## 3. Cross-Sectional Validation

Since the time span is not long enough, use the cross-sectional dimension, i.e. the number of coins, to compensate.

Logic: if your logic, such as RSI oversold + breakout, captures a universal market sentiment pattern, then it should not only work on BTC.

Method:

Find the best `thresh` using ETH historical data.

Directly apply this parameter to backtests on SOL, BNB, or BTC.

Judgment: if the parameter fitted on ETH performs terribly on SOL, it means this parameter is “overfitted” to ETH’s specific price movement rather than capturing a general trading logic.

Note: choose coins with relatively lower correlation, although the crypto market is generally highly correlated, or choose different sectors, such as public chains vs DeFi vs Meme.
